# Proposal 032: cards.json 역반영과 카드 미리보기 가독성 개선

## 배경

사용자가 `src/data/cards.json`의 카드 데이터를 직접 수정했다.

이 변경을 `docs/cards/onboarding/`의 카드별 원천 문서에 역반영해야 한다.

또한 카드 미리보기에서 효과 문구가 3줄 이상일 때 잘리고, 효과 안의 키워드 표기가 아직 텍스트 그대로 보여 카드 가독성이 떨어진다.

## 목표

1. `src/data/cards.json` 변경분을 기준으로 카드별 원천 문서에 범위/병력/효과/키워드 변경을 역반영한다.
2. 카드 효과 텍스트 영역을 약 10px 더 크게 잡는다.
3. 필요하면 일러스트 영역을 조금 줄여 효과 텍스트가 덜 잘리게 한다.
4. 효과 문구 안의 백틱 표기, 예: `` `기마` ``를 일반 텍스트 또는 칩 표기로 정리한다.
5. 효과 문구 안의 대괄호 키워드, 예: `[자폭]`, `[단말마]`, `[출정]`을 스크린샷처럼 칩 형태로 렌더링한다.
6. 장악범위/공격범위가 없는 카드도 범위 그리드와 사람 아이콘을 표시한다.

## UI 수정 방향

### 효과 영역

현재 `src/App.tsx`의 카드 효과 문구는 다음 계열로 렌더링된다.

```tsx
<p className="line-clamp-3 ...">
  {card.effect}
</p>
```

제안:

- 효과 박스 높이를 약 10px 늘린다.
- `line-clamp-3`을 제거하거나 `line-clamp-4`로 바꾼다.
- 일러스트 영역 높이를 조금 줄여 하단 효과 영역과 겹치지 않게 한다.

### 키워드 칩

효과 텍스트 안의 `[키워드]`는 칩으로 렌더링한다.

예:

```text
[자폭] 공격 범위 내에 상대의 병사가 있다면...
```

렌더링:

- `[자폭]` 부분은 둥근 테두리/배경을 가진 작은 인라인 칩.
- 나머지 문장은 일반 텍스트.

백틱 표기는 화면에서 제거한다.

예:

```text
`기마` 타입 유닛과 교전 전...
```

렌더링:

- 백틱 없이 `기마 타입 유닛과 교전 전...`
- 만약 백틱 안의 단어도 칩으로 처리할지 여부는 확인 필요.

### 범위 그리드

장악범위나 공격범위가 없는 카드도 그리드와 사람 아이콘을 보여준다.

현재 데이터에서 `없음` 대신 `0`을 넣으면 의도한 대로 처리될 가능성이 있다.

구현 방향:

- `movement`, `attackInfluence`가 비어 있거나 `없음`이어도 `RangeGrid`를 렌더링한다.
- 활성 칸은 없게 둔다.
- 사람 아이콘은 `controlIconPosition`, `attackIconPosition`이 있으면 그 위치에 표시한다.
- `0`은 “활성 칸 없음”으로 해석한다.

## 문서 역반영 방향

`src/data/cards.json`에서 변경된 필드를 카드명 또는 id 기준으로 찾아 `docs/cards/onboarding/` 문서에 반영한다.

대상 필드:

- `movement` -> `장악범위`
- `controlIconPosition` -> `장악범위 사람 아이콘`
- `attackInfluence` -> `공격범위`
- `attackIconPosition` -> `공격범위 사람 아이콘`
- `capacity` -> `병력`
- `effect` -> `효과`
- `keywords` -> `키워드`
- `species` -> `종족`
- `typeLabel`, `typeSpeciesLabel`은 필요 시 문서의 타입/종족 표기 보정에 사용

## 확인 질문

1. 효과 안의 백틱 단어, 예: `` `기마` ``도 칩으로 만들까, 아니면 백틱만 제거하고 일반 텍스트로 둘까?
2. `[자폭]`, `[단말마]` 같은 키워드 칩 색상은 스크린샷처럼 키워드별로 다르게 둘까, 아니면 일단 공통 스타일로 통일할까?
3. 범위가 없는 카드는 `0`을 “활성 칸 없음”으로 확정해도 될까?
4. 카드 효과가 그래도 길면 글자 크기를 줄여서라도 전부 보이게 할까, 아니면 4줄까지만 허용할까?
5. `cards.json` 변경분 역반영은 모든 카드 문서에 바로 적용할까, 아니면 이번에 사용자가 수정한 범위/공격범위 관련 필드만 먼저 반영할까?

## 진행 방식

사용자가 `go`라고 말하면:

1. `task.csv`에 새 작업을 추가한다.
2. `git diff src/data/cards.json` 기준으로 변경된 카드 데이터를 확인한다.
3. 카드별 원천 문서를 갱신한다.
4. `src/App.tsx`의 카드 효과 렌더링을 칩 지원 구조로 바꾼다.
5. 효과 영역과 일러스트 영역 위치를 조정한다.
6. 빈 범위도 그리드와 사람 아이콘이 나오도록 `RangeGrid` 처리 로직을 수정한다.
7. `pnpm run build`를 실행한다.
8. `localhost:5173/goe/cards`에서 긴 효과/키워드 칩/빈 범위 그리드를 확인한다.
